# Требования заказчика — версия 2

**Дата:** 12.04.2026
**Проект:** Textile Workshop ERP
**Источник:** Обратная связь от владельца швейного цеха

---

## 1. Локализация (переводы)

**Проблема:** Сейчас переводы на русском и кыргызском содержат ошибки и пропуски. Часть интерфейса отображается на английском или с некорректным текстом.

**Требование:** Полный корректный перевод всего интерфейса на три языка — русский, кыргызский, английский. Без ошибок, без пропущенных строк. Все кнопки, заголовки, плейсхолдеры, сообщения об ошибках, статусы заказов — всё должно быть переведено.

**Приоритет:** Высокий

---

## 2. Производство — убрать привязку к сотрудникам

**Контекст:** Швеи работают цепочкой (конвейер) — одна шьёт рукав, другая воротник. Нет смысла привязывать отчёт к конкретному сотруднику.

**Требование:**
- Убрать поле «Сотрудник» из формы создания отчёта производства
- Отчёт привязывается только к заказу и дате
- Отображать два вида отчётов:
  - **За сегодня** — что сшили сегодня (по заказам)
  - **За месяц** — сводка за выбранный месяц (по заказам)
- Фамилии сотрудников в разделе «Производство» не нужны

**Приоритет:** Высокий

---

## 3. Поиск по сотрудникам

**Проблема:** При большом количестве сотрудников приходится листать весь список, чтобы найти нужного человека.

**Требование:** Добавить строку поиска в списке сотрудников — поиск по имени в реальном времени (фильтрация при вводе).

**Приоритет:** Средний

---

## 4. Два типа зарплаты: оклад и сделка

**Контекст:** Не все работники на сделке. У некоторых фиксированный оклад.

**Требование:**
- При добавлении сотрудника выбирать **тип оплаты**:
  - **Оклад** — фиксированная сумма в месяц (вводится вручную администратором)
  - **Сделка** — оплата за количество изделий (ставка × количество сшитого)
- Для сотрудников на сделке: при вводе ежедневного отчёта система автоматически считает заработок за день
- Для сотрудников на окладе: администратор вводит сумму вручную
- В таблице зарплат показывать оба типа с итоговой суммой

**Приоритет:** Высокий

---

## 5. Проверка остатков при создании заказа

**Контекст:** При записи нового заказа указывается, сколько ткани/фурнитуры нужно на одно изделие. Система должна проверять, хватает ли материалов.

**Требование:**
- При создании заказа: если `количество × расход на единицу > остаток на складе` — показать предупреждение
- Предупреждение не блокирует создание заказа (заказ можно создать заранее), но явно информирует:
  _«Внимание: для этого заказа не хватает материала "Кашемир Премиум" — нужно 25м, на складе 4.2м»_
- Проверка выполняется по техкарте (Bill of Materials) выбранного продукта

**Приоритет:** Высокий

---

## 6. Уведомления (колокольчик)

**Проблема:** Иконка колокольчика в шапке не работает — нельзя нажать и посмотреть уведомления.

**Требование:**
- При нажатии на колокольчик открывается выпадающее окно со списком уведомлений
- Типы уведомлений:
  - Материал ниже минимального остатка (с названием и текущим количеством)
  - Заказ приближается к дедлайну (за 3 дня)
- На колокольчике — красный кружок с количеством непрочитанных уведомлений
- Уведомления можно отметить как «прочитанные»

**Приоритет:** Средний

---

## 7. Смена пароля и данных администратора

**Проблема:** Нет возможности сменить пароль или обновить данные профиля через интерфейс.

**Требование:**
- Добавить страницу **«Профиль»** или **«Настройки»**
- Возможности:
  - Сменить пароль (текущий пароль + новый пароль + подтверждение)
  - Изменить ФИО, телефон
- Доступ к странице — через аватар/имя в правом верхнем углу

**Приоритет:** Средний

---

## 8. Исправление ошибок — редактирование и удаление данных

**Контекст:** Если администратор случайно ввёл неправильное имя, цену или другие данные, нужна возможность это исправить.

**Требование:**
- **Все сущности** должны поддерживать редактирование и удаление:
  - Клиенты — есть ✅
  - Заказы — есть ✅
  - Материалы — есть ✅ (добавить удаление)
  - Сотрудники — добавить полное редактирование (включая ставку, тип оплаты)
  - Отчёты производства — добавить редактирование и удаление
  - Продукты — добавить страницу управления продуктами в интерфейсе (сейчас только через API)
- При удалении — всегда подтверждение: _«Вы уверены? Это действие нельзя отменить»_
- Если сущность используется в других записях (например, клиент с заказами) — предупредить перед удалением

**Приоритет:** Высокий
